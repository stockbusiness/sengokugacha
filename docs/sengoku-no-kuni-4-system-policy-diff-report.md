# 「千ノ国 4システム共通認識・連携方針書v2.1」現行実装差分報告

本ドキュメントは、アップロードされた「千ノ国 4システム共通認識・連携方針書v2.1」(以下「方針書」)の指示に基づき、**改修は行わず**、現行実装(本リポジトリ=戦国パスポート)との差分・影響範囲・必要API・データ移行の要否を調査した報告書である。各システムの具体的な実装範囲は、本報告の確認後に個別指示書で確定する前提のため、本書では実装方針の提案は最小限に留める。

---

## 0. 総論: 現行アーキテクチャの位置づけ

方針書は「代理店システム」「千ノ国パスポート」「ショッピングシステム」「千ノ国ウォレット」の4システムを、`common_user_id`を軸にした共通顧客HUB経由で疎結合に連携させる構想である。

現行の本リポジトリは、この4役割を**単一のNext.js+Supabaseモノリス内で兼務**している。

| 方針書のシステム | 現行実装での相当箇所 |
|---|---|
| 千ノ国パスポート | 本リポジトリのLIFFアプリ本体(`src/app/(app)/*`) |
| 代理店システム(業務機能) | `agents`テーブル+`/agency/*`(代理店ポータル)+外部システム`sengoku-ai.com`との同期(`agency-sso.ts`, `agency-integration`) |
| 代理店システム(共通顧客HUB) | **存在しない**。`users.referring_agent_id`という単一カラムが代替している程度 |
| ショッピングシステム | `purchases`テーブル+Stripe webhook(石高/ガチャ券/城主プラン)、および完全に別系統の`external_orders`(外部購入管理) |
| 千ノ国ウォレット | **台帳としては存在しない**。`users.kokudaka`/`users.gacha_tickets`/`users.contribution_points`という残高カラムの直接増減のみ |

方針書が要求する「システム分離+共通ID+イベント連携」という構造そのものが、現行実装には存在しない。したがって差分は「個別機能の過不足」ではなく、**アーキテクチャ全体の設計思想の差**という性質が大きい。

## 1. 概念ごとの差分

### 1-1. 共通ユーザーID(方針書7章・24章)

| `system_account_links`(各システムID対応表) | 存在しない | **未実装** |

## 2. 影響範囲

方針書をそのまま適用する場合、影響が及ぶ主な既存資産は以下の通り。

- **`users`テーブルとその参照箇所全て**(約30箇所以上): `id`を`common_user_id`相当に格上げするか、別テーブルに`common_user_id`を新設して`users`をパスポート固有プロフィールへ格下げするかで、影響範囲が大きく変わる。
- **`agents`テーブルと代理店ポータル・SSO**: `agents.user_id`追加は比較的小さな変更だが、「代理店本人も一般ユーザー」という前提の徹底は、代理店ログインセッション(`agency-sso.ts`)と一般ユーザーセッション(LIFF)の統合方式に関わる。
- **`purchases`テーブルと関連API**: 販売帰属スナップショット列の追加、`item_type`ごとに分散した特典付与ロジック(`grantPurchase()`等)の統一。
- **`external_orders`テーブル**: 既に指示書v1.0で「代理店報酬計算には使わない」と明記した設計判断があり、方針書の「販売担当・クロージング担当」概念とどう整合させるかは要調整。
- **`users.kokudaka`/`gacha_tickets`/`contribution_points`**: 台帳化する場合、残高参照箇所(ホーム画面、ガチャ実行時の残高チェック、管理画面の一覧表示等、数十箇所)を全てSUM集計または最新残高キャッシュ参照に置き換える必要がある。
- **`agent_sales`テーブルと代理店報酬計算**: 3系統(石高/ガチャ券用`agent_sales`、城主プラン用`commission_ledger`、外部購入用=報酬計算なし)に分かれている報酬ロジックの統一。
- **既存の「城主プラン」「国取り連携」「外部購入管理」の各実装**: これらは全て「方針書以前」に設計されたものであり、特に外部購入管理は「代理店報酬は外部システムが正」という前提で意図的に代理店連携を薄くしている。方針書の統合方針とは設計思想がやや対立する部分がある。

## 3. 必要API(方針書27章との対比)

方針書が例示する論理APIのうち、現行実装に相当するものが無いため、**全て新設が必要**。

| 方針書の論理API | 現行での近似実装 | 新設要否 |
|---|---|---|
| `POST /users/resolve` | 無し(LINEログイン時に`findOrCreateUserByLineId()`が内部関数として同等処理をしているが、外部公開APIではない) | 要新設(または既存関数の外部API化) |
| `POST /users` | 無し | 要新設 |
| `GET /users/{common_user_id}` | 無し(`/api/admin/users`はadmin検索専用) | 要新設 |
| `POST /users/{common_user_id}/accounts/link` | 無し | 要新設 |
| `POST /users/{common_user_id}/merge` | 無し | 要新設 |
| `POST /referrals/capture` | 無し(`sessionStorage`への一時保存のみ、API化されていない) | 要新設 |
| `POST /referrals/confirm` | 無し(`passport.ts`内のfindOrCreateUser処理に暗黙的に内包) | 要新設 |
| `POST /agents/assign` | 無し | 要新設 |
| `GET /users/{common_user_id}/sales-context` | 無し | 要新設 |
| `POST /orders/{order_id}/sales-context/confirm` | 無し | 要新設 |
| `POST /entitlements/grant` / `revoke` | 無し(webhook内に直書き) | 要新設(既存ロジックの抽出) |
| `POST /wallet/credits` / `debits` | 無し(`users`テーブル直接更新) | 要新設(台帳テーブルとセット) |

**現状は「本リポジトリが1システムで完結している」ため外部APIが不要だった**、という経緯を踏まえると、これらのAPI新設は「4システムを本当に別サービスとして分離する」という意思決定と不可分である。もし4システムを引き続き同一リポジトリ内で運用する場合でも、方針書の意図(将来的な分離可能性、監査性、冪等性)を満たすには、内部関数呼び出しをこれらのAPI相当の関数インターフェースとして明確に切り出す設計変更が必要になる。

## 4. データ移行の要否

**移行が必要になる可能性が高い項目:**

1. **`common_user_id`の新規発行**: 既存`users.id`をそのまま`common_user_id`として採用するか、別ID体系にするかで移行要否が変わる。前者なら移行コストは低いが、代理店(`agents`)を「共通ユーザー」として遡及登録する場合は、既存代理店全件に対応する`users`行の新規作成または紐付け作業が発生する(方針書30.2「自動統合しないケース」に注意しながら)。
2. **登録経路(`acquisition_channel`)の遡及付与**: 既存ユーザーには登録経路の記録が無いため、過去分は「不明(unknown)」等のデフォルト値で埋めるしかなく、正確な遡及は不可能。
3. **販売帰属の4区分への分解**: 既存の`purchases.selling_agent_id`・`agent_sales`・`external_orders`の各スナップショットを、`registration_referrer_agent_id`/`assigned_agent_id`/`order_sales_agent_id`/`order_closer_agent_id`にどう振り分けるかのマッピングルールが必要。過去データは情報が欠落しているため、正確な事後分類は困難な項目がある。
4. **残高カラム→台帳への移行**: `users.kokudaka`/`gacha_tickets`/`contribution_points`の現在値を、台帳の「初期付与(migration)」行として1件ずつ起票する形で移行可能(残高自体は保存されているため、金額の整合性は保てる)。ただし過去の個別付与・使用履歴(いつ何のために増減したか)は、`user_activity`テーブルにある分を除き、遡及復元できない。
5. **代理店の階層・報酬確定状態**: `commission_ledger`は方針書の台帳方式とほぼ整合しているため、移行不要かごく軽微な調整で済む可能性が高い。`agent_sales`(石高/ガチャ券用)は確定/取消の概念が薄いため、移行時に「これまでの実績は全て`confirmed`扱いとして扱う」等の一括ルールが必要になる可能性がある。

**移行が不要と考えられる項目:**

- 城主プランの`commission_ledger`/`commission_adjustments`(既に台帳方式・バージョン管理があるため、方針書と親和性が高い)
- `external_orders`の状態遷移マトリクス(方針書のイベント一覧と直接の対応はないが、既存の12状態モデル自体は保持可能)

## 5. 既存不具合(方針書とは別件、調査中に発見)

方針書との差分ではないが、今回の調査(3-5節)で以下の既存の実装漏れを発見したため、参考として報告する。

- **返金時、石高・ガチャ券購入分の`agent_sales`実績が取り消されない**(`/api/admin/purchases/[id]/refund/route.ts`)。land_plot購入の返金は`commission_ledger`への反対仕訳が作られるが、石高・ガチャ券の返金にはそれに相当する処理が無く、代理店の売上実績・報酬対象額に返金後も残り続ける。
- **国家貢献ポイント(`contribution_points`)は返金時に一切取り消されない**。
- 同一`purchases`行への返金APIの二重実行防止は`status`ガードのみで、DBレベルの排他制御・トランザクションが無いため、理論上は同時2リクエストでの競合の余地が残る(実害の報告は無いが、方針書29.1「冪等性」の観点でも指摘に値する)。

これらは方針書の適用有無にかかわらず対応可否をご判断いただける独立した課題のため、修正が必要であれば別途指示をお願いします。

## 6. 未回答のまま残る論点(現状確認のみでは判断できない事項)

- 「代理店システム」「ショッピングシステム」「千ノ国ウォレット」を、今後も本リポジトリと同一プロジェクト内で実装するのか、それとも別リポジトリ・別サービスとして分離するのか(方針書のAPI連携前提は後者を強く示唆しているが、本書はその決定を含んでいない)。
- 「千ノ国パスポート」は「戦国パスポート」のリブランドなのか、それとも戦国の国はその中の1コンテンツという位置付けになるのか(方針書2.1「戦国の国のゲーム、ガチャ、カード収集」という表現から、後者の可能性が読み取れる)。
- 既存の`agents`テーブル・代理店ポータルをそのまま「代理店システムの業務機能」として存置し、新設する「共通顧客HUB」だけを追加する増分アプローチを取るか、`agents`テーブル自体を再設計するか。

これらは指示書v1.0・collection-conquest指示書と同様、個別指示書確定時に方針を明示いただければ、それに沿って実装計画を立てます。
